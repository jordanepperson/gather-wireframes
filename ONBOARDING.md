## Design system (shared across pages)

- `css/tokens.css` — colors (`--color-off-white #FFFAF0`, `--color-bright-blue #2A5ED8`, `--color-dark-blue #12161D`, `--color-light-blue #BDD8E7`), type scale, spacing scale, and four `.pairing-01`–`.pairing-04` utility classes (background/heading/body/accent/button color sets). **`.pairing-03`'s `--heading` was changed from bright-blue to off-white this session** (dark-mode headings read better in white; eyebrows stay bright-blue via their own explicit class color, unaffected) — this is a shared token change, so it affects every page using `pairing-03`, not just Gather Global.
- `css/base.css` — reset, layout, buttons, motif shapes (`.motif-ripple`, `.motif-circle`, `.motif-dot-globe`, `.motif-constellation`), the floating back-to-index button, `overflow-x: hidden` on `html`/`body`.
- Fonts: Plus Jakarta Sans (primary) + Instrument Serif (secondary/accent, italic by default — regular weight is IF:Gathering-only per brand doc), loaded via Google Fonts CDN.
- **Known gotcha #1**: centering an eyebrow/small-caps label needs both `text-align:center` AND `margin-inline:auto` — the global `p { max-width: 60ch }` rule shrinks the element's own box down, so without the auto margin it centers text within a box that itself isn't centered.
- **Known gotcha #2 (bit twice this session)**: a white/off-white card nested inside a dark `pairing-03` section silently inherits `--body`/`--heading` custom properties from the ambient section (they cascade through the DOM regardless of the card's own background), producing invisible off-white-on-white text or invisible buttons (`.btn-outline` uses `color:inherit`/`currentColor`). Any time you put a light card inside a dark section, explicitly set `color` on its text and on any `.btn`/`.btn-outline` inside it — don't rely on inheritance.
- Logos in `assets/`: `gather-dark/light.svg` (parent wordmark), `gather-27-dark/light/blue.svg`, `gather-cities.svg` + `gather-cities-dark.svg` + `gather-cities-light.svg`, `if-gathering-dark.svg` + `if-gathering-light.svg`, `gather-cluster-dark.svg` + `gather-cluster-light.svg` (a combined-brand lockup mark, added mid-session). **Dark-suffix = dark-colored mark for light backgrounds; light-suffix = light/white-colored mark for dark backgrounds.** All four pages now have both variants available for every mark, so the full sub-brand logo row can go in any footer regardless of background color — check which variant matches the section's `--bg` before using one.

## Patterns built (shared across all 4 pages now)

- **Tooltip system — rebuilt this session, now `data-tooltip`-attribute-based** (was previously an `href`-keyed `DESTINATIONS` lookup restricted to `a.btn`). Current version: one script per page queries `document.querySelectorAll('[data-tooltip]')` and shows a "Goes to: {label}" pill following the cursor on hover. Works on any element — buttons, plain anchors, even non-link `<div>`s (e.g. story/moment cards). **Governing convention, per explicit client direction on both Gather Everywhere and Gather Global: every button/link that has a `data-tooltip` is annotation-only and must have `href="#"` — it does NOT actually navigate anywhere.** This was a deliberate choice to stop reviewers from click-navigating during demos; it also means several links that used to be *real* in-page anchors (e.g. "Our Story" → `#founder`, nav "About"/"Event"/"FAQ" → real section anchors) were intentionally converted to `href="#"` + a tooltip showing the *intended future* path (e.g. `/about`, `#the-event`, `/Gather27`). The one deliberate exception on Gather Everywhere: the hero's "Gather With Us" button has **no** tooltip and still does a real in-page scroll, because it's genuinely useful in-page navigation, not a placeholder. When extending this pattern to Gather Cities/IF:Gathering, follow the same rule — don't assume every button needs a tooltip; ask if it's ambiguous whether a given link should stay "real" or go tooltip-only. `gather-cities.html` and `if-gathering.html` got the tooltip CSS (`.btn-goto-tooltip`) and script added this session (they didn't have it before) even though their content hasn't had a full tooltip audit yet.
- **Unified footer lockup**, now on all four pages: cluster logo (dark/light matching bg) → sub-brand logo row (Gather Global / Gather Cities / IF:Gathering, tooltip-only, no routing) → "[Brand] is part of Gather, a movement founded by Jennie Allen" line → Privacy/Terms row. Order matters and was explicitly requested: Privacy/Terms goes **below** the Gather disclaimer line, not above it (this was a correction — check this order if you're templating the footer onto Cities/IF).
- **Nav hamburger dropdown** (`js/nav-menu.js`) — still only wired up on `gather-everywhere.html`. Its "The Movement" group order is Our Story → Our Partners → Jennie Allen (Jennie Allen deliberately last, per explicit request).
- **Content-note pattern for placeholders**: visible small italic caption baked into the placeholder (not a hover tooltip) — for whoever sources the real photo/video later. Used throughout (hero media, art-moment, pillar cards, Moments reel cards, etc).
- **Real rotating globe**: d3-geo orthographic projection + `world-atlas` topojson data via jsdelivr CDN (`d3@7`, `topojson-client@3`, `world-atlas@2/countries-110m.json`). On Gather Global, the globe surface itself is dark (`--color-dark-blue`), dots are solid off-white (no stroke), and **each hub dot has a continuously-updating on/off-screen label** (an HTML div positioned via the same per-frame projection math, faded in/out based on visibility) that names the hub as it rotates into view — replaced an earlier hover-tooltip-per-dot approach that the client found unusable on a spinning globe. Respects `prefers-reduced-motion`.
- **Modal pattern** (Gather Global): reusable `.modal-overlay`/`.modal-box` + `data-modal-open`/`data-modal-close` JS, used for the Pray/Worship/Repent pillar-card modals and for a shared, dynamically-populated leader-bio modal (reads `data-name`/`data-location`/`data-bio`/`data-social` off whichever leader card was clicked). The persecuted-church leader entry ("Ashok") deliberately shows "Social links withheld for security" instead of social links — a real safety consideration, mirrored from how gather25.com itself handles persecuted-church speakers (first name only, no location specifics).
- **Leader pagination**: simple JS, 4 leaders per page, prev/next + dot indicators, `.leader-card.is-hidden` toggling. 13 leaders total (12 named + 1 persecuted-church).
- **Collapsed accordions**: About Gather Global's three theme callouts are `<details>`/`<summary>` elements (heading text as the summary), collapsed by default.
- **Moments reel** (Gather Global's Stories section, styled after passion2027.com's video-reel pattern): horizontally-scrolling row of 9:16 vertical cards with a centered play-button icon, gradient placeholder + content-note, and left/right arrow buttons. **Known tool-specific bug worth knowing about**: `Element.scrollBy({behavior:'smooth'})` does not animate at all in this session's Browser-pane environment (confirmed via direct testing — `behavior:'auto'` works instantly, `'smooth'` never moves the scroll position even after a 2s wait, and it's not a `prefers-reduced-motion` issue). The arrow buttons use `behavior:'auto'` for this reason. Untested whether this is a real cross-browser issue or purely a tool quirk — if a user ever reports the arrows not scrolling in their actual browser, that'd be worth a closer look, but it works fine as instant scroll either way.
- **Review Mode** (section-by-section feedback UI) exists in `js/review-mode.js` + `css/review-mode.css` but is **currently disabled** — `var ENABLED = false;` at the top of `js/review-mode.js`. Flip that one flag back to `true` to re-enable it everywhere.

## Working style notes

- User wants everything **verified in the browser**, not just claimed done — screenshot or JS-assertion proof before saying something works.
- Fast, iterative, high-volume requests — batches of 5-10 discrete asks per message are normal; work through them as a list, verify at the end, one commit + push per batch (not per line-item).
- Push cadence: push after essentially every verified commit — established explicitly as the norm since two sessions share this repo.
- Silently fix obvious typos in requested copy rather than reproducing them literally.
- When a design/content question is genuinely open (not just "implement X"), give a short recommendation + the main tradeoff and ask before committing — used `AskUserQuestion` successfully for e.g. "keep the Repent pillar or cut it?" (kept it — it's structurally part of the real Gather25/27 four-movement format, confirmed via gather25.com's own FAQ language, not a stray addition) and "what to do about missing logo assets?" (user chose to wait for the real assets rather than a text fallback — and then supplied them a couple messages later, at which point the deferred work was completed proactively).
- The user is sensitive to copy that reads as AI-generated — was asked explicitly at one point to strip em dashes and "it's not X, it's Y" / "this is X, not Y" rhetorical constructions and make copy "sound more like a church website." Worth keeping in mind for any new copy you write on this project generally, not just where explicitly flagged.
- When a brand asset is missing (e.g. a logo color variant that doesn't exist yet), flag it and ask rather than fabricating placeholder brand art.
- Occasionally the exact same user message arrives twice in a row (a client/transport quirk, not a real repeat request) — if you find the described work already fully implemented and verified, don't redo it; just re-verify and report status.

## Known Browser-pane tool quirks (this session)

These are tool/environment limitations observed repeatedly, **not** bugs in the wireframe code — verify via DOM/`getComputedStyle`/JS assertions when you hit them rather than assuming the underlying markup is broken:
- Screenshots reliably render correctly near the top of a page, but on long pages, scrolling deep (roughly 700px+) and then screenshotting frequently returns a solid-black/blank frame even though the live DOM is correct (confirmed repeatedly by cross-checking `getBoundingClientRect()`, computed styles, and `document.elementFromPoint()` at the same scroll position — the real layout is fine, the screenshot capture itself is stale/broken at that scroll depth).
- `window.innerWidth`/`innerHeight` occasionally corrupt to `0` on a tab after enough interactions (especially after a `resize_window` call) — everything measures as zero/degenerate until you open a fresh tab. If measurements look impossible (e.g. a flex child's rect shows 0 width, or heights don't add up), check `window.innerWidth/innerHeight` first before concluding the CSS is wrong.
- `scrollBy({behavior:'smooth'})` does not animate in this environment (see Moments reel note above) — use `'auto'` for anything you need to verify actually moved.
- Workaround pattern that's worked well all session: do the visual/structural verification via `mcp__Claude_Browser__javascript_tool` (computed styles, `getBoundingClientRect`, live click simulation) as the primary source of truth, and treat screenshots as a nice-to-have secondary confirmation rather than the arbiter of correctness.
