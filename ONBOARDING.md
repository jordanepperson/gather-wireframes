## Design system (shared across pages)

- `css/tokens.css` — colors (`--color-off-white #FFFAF0`, `--color-bright-blue #2A5ED8`, `--color-dark-blue #12161D`, `--color-light-blue #BDD8E7`), type scale, spacing scale, and four `.pairing-01`–`.pairing-04` utility classes (background/heading/body/accent/button color sets). **`.pairing-03`'s `--heading` was changed from bright-blue to off-white** (dark-mode headings read better in white; eyebrows stay bright-blue via their own explicit class color, unaffected) — a shared token change affecting every page using `pairing-03`. Gather Cities additionally carries its **own page-scoped override** of `.pairing-01`'s heading color (see above) — that one lives in `gather-cities.html`'s own `<style>` block, not in `tokens.css`, so it doesn't leak to other pages.
- `css/base.css` — reset, layout, buttons, motif shapes (`.motif-ripple`, `.motif-circle`, `.motif-dot-globe`, `.motif-constellation`), the shared `.site-utility-bar` + notes-modal styles (see dedicated section above). **Gotcha**: `section { padding-block: var(--space-7); }` is a global default — any full-viewport-height hero section needs to explicitly zero this out (see Gather Cities' `#hero { padding-block: 0; }`) or it'll throw off height/centering math.
- **Real bug fixed this session, worth understanding**: `html` and `body` both had `overflow-x: hidden` with no explicit `overflow-y` — a CSS interop rule silently computes `overflow-y` to `auto` on an element whenever `overflow-x` is non-visible and `overflow-y` was left at its default `visible`. That made `body` register as its own scroll container, which hijacked `position: sticky` children (the new `.site-utility-bar`) into sticking relative to `body`'s box instead of the real viewport — and since `body`'s height always exactly matches its content, nothing ever visibly "scrolled" within body's own frame, so the bar never appeared to stick. **Fix: `body` no longer repeats `overflow-x: hidden` — only `html` has it now** (confirmed via `document.scrollingElement`, which is `html`). Verified this doesn't reintroduce a real horizontal-scroll regression on any page (`document.documentElement.clientWidth === window.innerWidth` everywhere checked). If you ever need to re-add `overflow-x: hidden` to `body` for some reason, also explicitly set `overflow-y: visible` won't actually help (the interop rule overrides it regardless) — the only real fix is not putting `overflow-x: hidden` on both `html` and `body` at once.
- Fonts: Plus Jakarta Sans (primary) + Instrument Serif (secondary/accent, italic by default — regular weight is IF:Gathering-only per brand doc), loaded via Google Fonts CDN.
- **Known gotcha**: centering an eyebrow/small-caps label needs both `text-align:center` AND `margin-inline:auto` — the global `p { max-width: 60ch }` rule shrinks the element's own box down, so without the auto margin it centers text within a box that itself isn't centered.
- **Known gotcha**: a white/off-white card nested inside a dark section silently inherits `--body`/`--heading` custom properties from the ambient section (they cascade through the DOM regardless of the card's own background), producing invisible off-white-on-white text or invisible buttons (`.btn-outline` uses `color:inherit`/`currentColor`). Any time you put a light card inside a dark section (or vice versa), explicitly set `color` on its text and on any `.btn`/`.btn-outline` inside it — don't rely on inheritance. Related: a card can also visually disappear if the *ambient section* and the *card* end up the same color after a palette change (happened when Gather Cities flipped to an off-white page background) — add an explicit border to any nested "card" component whenever the surrounding palette changes.
- **Known gotcha**: `display:flex` on a container with a single block-level child (e.g. `.container`, which has its own `margin-inline:auto`) makes that child shrink-to-fit and center itself instead of filling the row — give it an explicit `width:100%` if you actually want it to behave like normal block flow within a flex parent used only for cross-axis (vertical) centering.
- **Known gotcha**: a decorative child with `z-index: -1` (meant to sit "behind" the real content) inside a parent that also has `overflow: hidden` can render as fully invisible AND fail to hit-test (`elementFromPoint` returns the parent, not the child, even with `pointer-events` re-enabled and opacity forced to 1) — confirmed on IF:Gathering's Get Involved badge row. Don't use negative z-index for this pattern. Instead, leave the decorative element at `z-index: 0` (default stacking) and explicitly give the real foreground content (e.g. the `.container` holding the text/cards) `position: relative; z-index: 1` so it stacks above — same visual result, no invisibility risk.
- Logos in `assets/`: `gather-dark/light.svg` (parent wordmark), `gather-27-dark/light/blue.svg`, `gather-cities.svg` + `gather-cities-dark.svg` + `gather-cities-light.svg` + `gather-cities-blue.svg` (added mid-session, a blue-colored variant for use directly on off-white backgrounds), `if-gathering-dark.svg` + `if-gathering-light.svg`, `gather-cluster-dark.svg` + `gather-cluster-light.svg` (a combined-brand lockup mark). **Dark-suffix = dark-colored mark for light backgrounds; light-suffix = light/white-colored mark for dark backgrounds; blue-suffix (Cities only so far) = the bright-blue-colored mark, for use directly on off-white/light backgrounds where dark-suffix would look flat.** Check which variant matches the section's `--bg` before using one.
- **`if-badge.svg`** (added this session, dropped in as an untracked file before Jordan asked to actually use it): a circular "IF" medallion/seal, Light Blue fill (`#BDD8E7`), `viewBox="0 0 196 196"`. Currently used once, as a row of 5 faded copies behind IF:Gathering's Get Involved heading. If you resize it, note its viewBox height (196) already matches its width (square) — unlike `gather-27-light.svg`, no internal-alignment quirk to worry about with this one.
- **Gather Global's logo (`gather-27-*.svg`) has a real optical-alignment quirk**: its viewBox is taller than the wordmark's own ink (extra empty space at the bottom, reserved for the circular "27" badge), so when constrained to a fixed height alongside the other three logos, it visually sits noticeably higher. Fixed in the shared utility bar with a small `transform: translateY(2px)` scoped to `img[src*="gather-27"]` — if you use this logo at small sizes elsewhere in a row with other logos, expect the same issue and the same fix.

## Working style notes

- User wants everything **verified in the browser**, not just claimed done — screenshot or JS-assertion proof before saying something works. In practice this session, computed-style/`getBoundingClientRect` JS assertions ended up being the *primary* verification method, not screenshots (see tool quirks below) — that's an acceptable substitute, just say so explicitly when you're relying on it instead of a visual screenshot.
- Fast, iterative, high-volume requests — batches of several discrete asks per message are normal; work through them as a list, verify at the end. **This session shifted toward one commit + push per discrete change** (not batched at the end) once deep in Cities-page iteration, especially for quick copy/headline swaps — match the user's pace: if they're firing off single small tweaks back-to-back, commit each one rather than batching.
- Push cadence: push after essentially every verified commit — since two sessions share this repo.
- Silently fix obvious typos in requested copy rather than reproducing them literally (e.g. a word-order slip like "gather the with world" → "gather with the world").
- When a design/content question is genuinely open, give a short recommendation + the main tradeoff and ask before committing. `AskUserQuestion` has worked well repeatedly — e.g. offering 2-3 headline/eyebrow options with a recommended pick and letting the user choose or redirect, rather than silently picking one. Headlines in particular got iterated on *a lot* this session (hero headline alone changed 4+ times) — that's normal back-and-forth for this client, not a sign something's wrong; just execute each swap cleanly and move on.
- The user is sensitive to copy that reads as AI-generated — strip em dashes and "it's not X, it's Y" / "this is X, not Y" rhetorical constructions, make copy "sound more like a church website." Also watch for copy that reads as too clerical/operational (a Gather Cities paragraph got flagged as "reading a little off" for being too much about internal process/percentages rather than warm and inviting) — lean personal and inviting over official-sounding, even when the underlying content is genuinely procedural (e.g. a sign-up process can still be framed warmly).
- When a brand asset is missing, flag it and ask rather than fabricating placeholder brand art — but reconstructing a *lockup pattern* (text + dot, matching an existing real asset's visual language) in CSS/HTML when no such asset exists at all (e.g. "Gather Brooklyn") is fine and was explicitly requested.
- Hypothetical/illustrative content (e.g. a list of cities "in the queue") is fine to invent when the user explicitly says so — but say so in the UI (a caption noting it's illustrative) unless/until the user says to drop the caveat. This one got added, then explicitly removed a few messages later — the user's call to make, not a default to assume either way.
- Occasionally the exact same user message arrives twice in a row (a client/transport quirk, not a real repeat request) — if you find the described work already fully implemented and verified, don't redo it; just re-verify and report status.
- **Git hygiene note to self**: this session repeatedly staged unrelated file changes together (e.g. `git add fileA.html fileB.html` then one commit covering both) when the intent was separate commits per logical change — the diffs were always correct, just the commit messages ended up describing only part of what was actually included. Stage and commit one logical change at a time, even when it means an extra couple of `git add`/`git commit` round trips.
- When the user reports something "isn't showing up" after a verified-and-pushed change, check three things in order before assuming the code is wrong: (1) `git log`/`git status -sb` to confirm the commit is actually pushed and matches `origin/main`, (2) which URL they're viewing (localhost vs. the Vercel deploy — the second session handles deploys, which can lag a push), (3) a browser cache on their end (suggest a hard refresh). In this session the code was correct every time; it was always caching (see below) or a deploy-lag question.

## Known Browser-pane tool quirks (confirmed again this session)

These are tool/environment limitations, **not** bugs in the wireframe code — verify via DOM/`getComputedStyle`/JS assertions when you hit them rather than assuming the underlying markup is broken:

- **Page scrolling can go completely dead for extended stretches** — `window.scrollTo()`, `window.scrollBy()`, and `element.scrollIntoView()` all silently no-op (`window.scrollY` stays `0` no matter what). Confirmed this is page-agnostic (happened even on a plain `index.html` load with zero custom JS), so it's a tool-session issue, not something in the page. No reliable fix found beyond waiting/retrying later in the session; when it's happening, fall back entirely to `getBoundingClientRect()`/`getComputedStyle()` for below-the-fold verification instead of trying to screenshot a scrolled view.
- **`window.innerWidth`/`innerHeight` (and by extension `getBoundingClientRect()` on elements sized relative to the viewport) occasionally corrupt to `0` or otherwise-impossible values** on a tab that's had many `navigate`/interaction calls. Confirmed repeatedly this session. **The fix that reliably works: open a brand-new tab (`tabs_create`) and navigate fresh there**, rather than continuing to reuse or re-navigate the same degraded tab. Always sanity-check `window.innerWidth` alongside any suspicious measurement before concluding the CSS itself is wrong.
- **Screenshot capture can be visually misleading relative to what JS reports for the same tab** — observed a screenshot appear to show a narrower/differently-laid-out page than `getBoundingClientRect()`/`window.innerWidth` confirmed was actually rendered at that moment. If a screenshot looks wrong (e.g. elements overlapping that shouldn't), cross-check with a JS measurement in the *same* tab before treating the screenshot as ground truth — and if in doubt, open a fresh tab and re-screenshot rather than trusting a screenshot from a tab that's been through a lot of `navigate` calls already.
- Screenshots reliably render correctly near the top of a page, but scrolling deep and then screenshotting can return a solid-black/blank frame even though the live DOM is correct (cross-check with `getBoundingClientRect()`/computed styles at the same scroll position).
- `scrollBy({behavior:'smooth'})` does not animate in this environment — use `'auto'` for anything you need to verify actually moved.
- Simulated `hover` actions (the `computer` tool's `hover` action) do not reliably trigger the CSS `:hover` pseudo-class (`element.matches(':hover')` came back `false` immediately after a hover call) — verify hover-state CSS by reading the actual stylesheet rules (`document.styleSheets`) rather than trying to screenshot a hovered state.
- **Workaround pattern that's worked well all session**: do the visual/structural verification via `mcp__Claude_Browser__javascript_tool` (computed styles, `getBoundingClientRect`, live click simulation, direct stylesheet inspection) as the primary source of truth, and treat screenshots as a nice-to-have secondary confirmation rather than the arbiter of correctness — especially now that scrolling itself can't be relied on.
- **Confirmed again, repeatedly, in a later session — the Browser pane caches static assets independently of the real server**, and it does NOT clear on a normal `navigate()`, a brand-new tab, or even a full `preview_stop`/`preview_start` restart. This isn't a one-time fluke tied to `css/base.css` specifically — it happened again on `js/notes-modal.js` in the same session, and the cache is keyed per-origin (i.e. per port): once you've loaded a file under `localhost:5178`, for example, that origin's copy is stuck even if you edit the file again and reload the same port. Symptom: you edit a shared CSS/JS file, verify via `curl` that the static server is serving the new bytes, but `getComputedStyle()` / DOM output in the Browser pane still reflects the old version. Don't waste time re-editing "broken" code that's actually fine — check with `Array.from(document.styleSheets).find(...).cssRules`, a raw `curl` of the file, or (for JS) counting expected output (e.g. `grep -c "page:" js/notes-modal.js`) to confirm whether it's a real bug or this cache. **The fix: start the static server on a brand-new port you haven't loaded anything from yet** (increment: 5174, 5175, 5176... each edited file needs its *own* first-load-on-this-port to bypass the cache, not just one port-switch per session) and navigate there. Restore the conventional `5173` and kill the scratch ports when you're done so the next session isn't surprised by stray processes.
- `tabs_create` can fail with "Could not open a new tab... tab cap reached" after enough tabs accumulate in a long session. Fix: `tabs_context` to list open tabs, then `tabs_close` a few of the old/unused ones before retrying `tabs_create`. You don't need a literal new tab to escape the asset cache above, either — navigating an *existing* tab to a brand-new port works exactly the same, since the cache is keyed by origin, not by tab.
